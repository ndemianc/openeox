# 3. Discussion and Planning

## 3.1 Work while Core is in public review

## 3.2 API workstream kickoff (Issue #196)

- Sergii Demianchuk asked whether formal API requirements had already been defined.
- Thomas Schmidt confirmed formal API requirements were not yet finalized and outlined two API usage directions:
  - Product-scoped retrieval of OpenEoX data.
  - Organization/ecosystem-scoped retrieval of OpenEoX statements.
- Sergii proposed starting with a REST/OpenAPI-first approach (with possible later consideration for protocols such as SNMP/MQTT).
- Thomas Schmidt created Issue #196 and requested Sergii to lead initial exploration.
- Sergii agreed to lead and document findings via the issue.

## 3.3 Product identification discussion (Issue #2)

- The TC revisited unique product identification and alignment with existing approaches.
- Thomas Schmidt reiterated that creating a brand-new global identifier scheme is out of scope for OpenEoX, while reuse/mapping of existing identifier approaches is in scope.
- The TC discussed alignment with Shell and references to CSAF product modeling.
- Sergii agreed to investigate TC54 product-definition references and comment findings on Issue #2.
- Thomas agreed to locate prior proposal material and add references in the issue thread.
